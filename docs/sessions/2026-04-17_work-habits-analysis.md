## 🎯 對你最有用的 3 個 Insight

### 1. 凌晨的黃金時段被用在「想」大於「做」
平均 476 分鐘/session 中，討論 + 反思 + 確認估計佔 60%+，實際 code push / 內容完成只佔末尾 5-10%。

→ 這不是問題，但值得自問：**你要的是「產出量」還是「決策確定性」？** 如果是前者，凌晨該多寫少想；如果是後者，那你已經在最佳化正確的變量。

### 2. 心理摩擦 >> 技能摩擦
你的「卡」不是「不知道怎麼做」，是「不確定該做什麼」。這解釋為什麼和 Claude 60% 接受建議、40% 挑戰——**你在用對話逼近自己的真實想法**，不是在找答案。

→ 如果想減少心理摩擦，session 開頭先花 5 分鐘寫 「今天唯一必完成的 1 件事是什麼」，而不是「今天想討論什麼」。

### 3. 你的「完成」定義可能和業界不同
「改好了，npm run dev 驗證通過」對你就是一個週期的 done。這和傳統「所有 TODO 打勾」的 done 不同。

→ 如果要 ship（像 6/1 上工這種硬 deadline），需要刻意升級「完成」的門檻標準。否則你會永遠停在「還可以再 polish」的狀態，而 polish 是無上限的。
